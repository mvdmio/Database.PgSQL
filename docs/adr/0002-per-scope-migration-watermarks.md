# Track migration watermarks per scope instead of globally

`DatabaseMigrator` decided which migrations to run by comparing each migration's timestamp identifier against a single **global** `MAX(identifier)` watermark over the whole `mvdmio.migrations` table. When two migrators run against the same database for different assemblies (e.g. `MyApp.Data` and `MyApp.Billing`), the assembly whose migrations carry *lower* timestamps falls below the other assembly's watermark and is **silently skipped** — not all migrations run. We decided to make the watermark **per scope**: every migration belongs to a **Scope** (default = its declaring assembly's simple name, overridable on `IDbMigration`), the `migrations` table records the scope of each row, and a migration runs when its identifier exceeds `MAX(identifier)` *for its own scope*. Scopes advance independently, so one assembly's timeline can no longer suppress another's. See [CONTEXT.md](../../CONTEXT.md) for the Scope / Identifier / Watermark vocabulary.

## Considered options

- **Per-scope `MAX(identifier)` watermark (chosen).** Each scope keeps its own high-water mark; a migration runs if it is ahead of its scope's mark. Preserves the existing "everything at or below the mark is already present" contract that schema-first bootstrap depends on.
- **Exact `(scope, identifier)` membership** (run any migration whose pair is not recorded). Rejected: schema-first bootstrap applies an embedded `schema.sql` and records only a single baseline row per scope to represent *all* migrations folded into that schema — those sub-baseline migrations have no individual rows, so a membership check would re-run every one of them against tables the schema already created. The watermark is the only model compatible with baseline rows.
- **Per-scope target for `MigrateDatabaseToAsync`.** Rejected for now: kept the existing `long` identifier as a **global ceiling** ("advance every scope up to this identifier"). Migrations are up-only (`IDbMigration` has no `DownAsync`), so a global forward ceiling stays coherent and avoids a heavier API nobody has asked for.
- **Per-scope empty-check for schema-first bootstrap.** Rejected for this release: `IsDatabaseEmptyAsync` / `ShouldApplySchemaAsync` stay **global**. Applying an embedded schema for a *second* scope against an already-populated database is therefore unsupported — schema-first works for the first scope to touch a fresh database, or for multiple assemblies bootstrapped by a single `DatabaseMigrator` instance (the integration-test scenario). A per-scope empty-check is a tracked follow-up.

## Consequences

- **Schema identity.** `PRIMARY KEY (identifier)` no longer models identity (two scopes may legitimately share an identifier), so it is dropped in favour of a named `UNIQUE (scope, identifier)` index. Identifier collisions across scopes do not occur in practice (all consumers are controlled, timestamps do not collide); if one ever did, the second insert fails loudly on the unique index rather than corrupting state.
- **Nullable scope during a transition.** `scope` is added `NULL`-able. New rows always carry a scope; legacy rows are filled by a **temporary backfill** that matches executed rows to discovered migrations by identifier and sets scope only where `NULL` (so concurrent runners each fill their own part). Rows no runner recognises stay `NULL`, emit one warning, and are excluded from every concrete scope's watermark — surfacing later as a migration error to resolve manually. The backfill is `[Obsolete]` and removed in the next major version, at which point `scope` becomes `NOT NULL` and the unique index is promoted to `PRIMARY KEY (scope, identifier)` via `ADD PRIMARY KEY USING INDEX`.
- **Schema-file header.** The header carries one `-- Migration version: <id> (<name>) [<scope>]` line per scope; a single generalised regex still reads the old scope-less single-line form. Old headers record a `NULL`-scope baseline that the backfill heals, so existing consumers need not regenerate `schema.sql` on upgrade.
- **Breaking changes (MAJOR bump).** `DatabaseMigrator` now requires an `ILogger<DatabaseMigrator>` (new dependency on `Microsoft.Extensions.Logging.Abstractions`) — source-breaking across all constructors. `ExecutedMigrationModel` and `SchemaFileMigrationInfo` gain a `Scope`; `SchemaFileParser.ParseMigrationVersion` and `SchemaExtractor.GetCurrentMigrationVersionAsync` now return collections.
- **Concurrency.** The existing session-scoped advisory lock (ADR 0001) still serialises runs, so the table upgrade (`ADD COLUMN` / `DROP CONSTRAINT` / backfill) is performed once at a time without additional coordination.
